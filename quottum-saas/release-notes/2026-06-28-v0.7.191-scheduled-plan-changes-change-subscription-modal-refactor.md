Activates the \`workspace_plan_changes\` infrastructure that shipped dormant in migration 077 (v0.7.181). The operator's "Change subscription" modal is now 3 fields + reason: pick a plan, pick a cadence, pick when it kicks in. Today applies immediately; future queues a pending change that an hourly cron applies on the day. Customer gets the right email at the right moment.

### Added

- **\`POST /api/admin/workspaces/[id]/schedule-plan-change\`** — single entry point for the modal. Branches on date: \`effective_at\` ≤ now()+60s grace applies immediately + fires \`billing.plan_changed\`; future queues a row in \`workspace_plan_changes\` + fires \`billing.scheduled_plan_change\` heads-up. Refuses Stripe-active workspaces with 409.
- **\`DELETE /api/admin/workspaces/[id]/scheduled-plan-changes/[change_id]\`** — operator cancels a pending change. Soft delete; 409s on already-applied or already-canceled.
- **\`POST /api/cron/apply-scheduled-plan-changes\`** — hourly janitor. Auto-cancels with \`canceled_reason='user_matched_target'\` when the workspace already moved to the target plan (policy decision #10), or \`workspace_on_stripe\` when state flipped between schedule and apply. Otherwise applies + stamps applied_at + fires \`billing.plan_changed\` + logs to admin_audit (\`applied_by_cron=true\`).
- **PendingChangeBanner** on the AdminWorkspaceSubscription card — yellow chip with target plan + effective date + a Cancel button. Same chip surfaces inside the modal as context so the operator doesn't double-schedule.
- **Migration 081** — pg_cron schedule \`quottum-apply-plan-changes\` at \`0 * * * *\` UTC. Verified active in \`cron.job\`.
- **\`sendPlanChangedEmail\`** and **\`sendScheduledPlanChangeEmail\`** helpers in \`billing-emails.ts\`.
- **WorkspaceTimeline merges \`admin_audit\` events** with lifecycle dates into one chronological stream. Audit entries use \`describeAdminAction\` so a billing checkout reads as **"Started Stripe Checkout · Pro · monthly · 14-day trial carryover"** instead of dumping Stripe IDs. Most recent at the top; a "Today" marker drops at the future↔past boundary.
- **Eager \`workspace_settings\` row at workspace creation** — \`/api/auth/signup-workspace\` and \`/api/workspaces\` now insert the row at creation time. Migration 080 backfilled the ~existing workspaces missing one. Operator console reads real defaults (\$\$USD · 12 months · 90·60·30\$\$) instead of all \`—\`.

### Changed

- **Change subscription modal** — Plan + Cadence + When + Reason. Removed Status / Trial ends / Renews-period-end inputs (they're derived state — picking them independently from plan + effective date was creating drift). Submit button label flips between "Apply plan change" (today) and "Schedule plan change" (future).
- **\`/admin/workspaces/[id]\` layout** — KPIs moved before Credit/Settings; Credit balance + Workspace settings paired in a 2-col grid; Members + Member activity paired in a 2-col grid; Workspace timeline pinned full-width at the bottom carrying everything that used to be in the standalone "Recent admin actions" card.
- **Operator console dates render in America/Los_Angeles.** New \`src/lib/admin-time.ts\` (\`ADMIN_TIMEZONE\`, \`formatAdminDate\`, \`formatAdminDateTime\`) swept across 20 admin files. End-user surfaces unchanged.
- **Postgres database default timezone** also pinned to America/Los_Angeles via \`ALTER DATABASE postgres SET timezone\`. Studio SQL queries now display in PT. pg_cron schedules unaffected (\`cron.timezone='GMT'\`, separate setting).

### Notes for testing

- New cron fires at the next \`:00\` UTC. First run on an empty \`workspace_plan_changes\` table is a no-op (logs \`{ checked: 0, applied: 0 }\`). Inspect via Studio: \`select * from cron.job_run_details where jobname = 'quottum-apply-plan-changes' order by start_time desc limit 5;\`
- Stripe-active workspaces are gated out of the schedule endpoint. Operator cancels Stripe first via the existing flow, then schedules.
- Beta gate unchanged: \`operator_subscription_actions_v179\` (operators already granted).
