Phase 2 polish on the payment-required UX. Originally asked at v0.7.194 spec time ("There should be a popup when first sign in after the plan has been changed. The user shouldn't be able to see or do anything until provide a payment method"), pragmatically descoped to a redirect for v0.7.194/195, now finally a proper modal.

### What changed

\`PendingUpgradeModal\` — full-screen overlay (z-[210], non-dismissible) shown on top of whatever dashboard page the user lands on. Replaces the previous silent redirect to \`/dashboard/upgrade?from=signup\`.

The dashboard underneath is dimmed but visible, so the user sees what they're missing — feels more urgent than landing on a fresh upgrade page. Hidden on \`/dashboard/upgrade\` itself so the user can complete Checkout without the modal in their way.

### Two-case adaptive copy

| State | Modal headline | CTA \`from=\` |
|---|---|---|
| Fresh paid signup, no Stripe yet | "Welcome to **Pro** — last step" | \`from=signup\` |
| Operator-scheduled upgrade applied | "Your workspace was upgraded to **Pro** on [date]" | \`from=scheduled_change\` |

Credit balance surfaces inline when present: "You have **\$X.XX** in credit — Stripe will apply it to your first invoice automatically." Primary CTA → \`/dashboard/upgrade\`. No "cancel and downgrade" button — operator-scheduled upgrades require support to revert.

### Cascades with v0.7.195

When an operator-scheduled upgrade applies LIVE (Bobby sitting on \`/dashboard/quotes\`, janitor fires the upgrade), v0.7.195's \`WelcomePopupPoller\` notices the workspace's \`welcome_announcement\` flip and calls \`router.refresh()\` within 30s. The layout re-renders, the modal appears on Bobby's screen, no navigation needed. Full operator-to-customer push in well under a minute.

### Cost note

Layout query is three parallel reads (latest applied \`workspace_plan_changes\` row, plan name, credit balance) ONLY on the gating condition. Cost is zero for the 99% of page loads that aren't in this state.

### No migration

Pure code change. Same gating condition that previously triggered the redirect; identical reach. Existing \`/dashboard/upgrade\` page banners still active for users who navigate there directly — modal + page banner are reinforcing, not redundant.

### Smoke test

1. Operator-schedule Bobby for a Pro upgrade ~5 days out
2. Force the janitor: \`select public.call_saas_cron('/api/cron/apply-scheduled-plan-changes');\` (or wait for the next \`:00\` UTC)
3. Bobby (already on \`/dashboard\` doing nothing) → within ~30s, sees the modal appear on his screen
4. Modal copy reads "Your workspace was upgraded to Pro on [date]"
5. Bobby clicks Add payment method → lands on \`/dashboard/upgrade?from=scheduled_change\` (modal hidden, page banner active)
6. Bobby completes Checkout → \`plan_status\` flips to \`active\` via the webhook → modal stops gating
