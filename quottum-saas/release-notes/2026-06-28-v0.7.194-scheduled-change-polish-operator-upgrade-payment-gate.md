Three batches in one release. Closes several gaps in the operator subscription-actions surface that v0.7.191-193 left open.

### User-facing scheduled-change visibility

- **Pending-change banner on \`/dashboard/settings/billing\`.** When an operator queues a future plan change, the customer now sees a sticky amber banner on the in-app billing page (was getting only the email, the in-app side stayed silent):

  > 📅 **Your plan switches to Pro on June 30, 2026.**  
  > Scheduled by support — no action needed from you. We'll email a confirmation when it happens.

  Display-only; can't be canceled from here (operator-scheduled, operator-canceled).

- **Cancel-scheduled email** (\`billing.scheduled_plan_change_canceled\`, migration 082). When the operator clears a queued change AFTER the heads-up email had already fired, the customer's last touchpoint was the "your plan WILL switch on DATE" email — and then the change silently didn't happen. The cancel endpoint now sends a follow-up so the loop closes.

### Operator surface polish

- **\`useConfirm\` modal across the operator surface.** The four destructive actions added in v0.7.191/192/193 (cancel scheduled change, clear trial history, cancel queued welcome popup, remove feature override) were using \`window.confirm()\` — system dialog with no platform consistency. Swapped to the platform's \`useConfirm\` modal. Each confirm button now echoes the action (\`Cancel scheduled change\`, \`Remove grant\`, etc.) and renders red via \`tone='danger'\`.

- **\`WorkspaceTimeline\` merges \`email_log\`** alongside lifecycle dates and audit events. Failed sends get a distinct \`MailWarning\` icon + danger color + the error message in the hint, so the operator can immediately spot what the customer didn't actually receive. Pulls workspace-scoped emails (\`quote.sent\` etc.) plus platform billing emails (\`workspace_id IS NULL AND recipient = workspace owner\`).

### Policy C — operator-scheduled upgrades require payment

Per the locked operator-subscription-policy memory, an operator-triggered upgrade is the user's commitment to pay — they shouldn't get a free trial dance on top of it, and they shouldn't be able to use the upgraded features until a card is on file.

- **Janitor** (\`POST /api/cron/apply-scheduled-plan-changes\`):
  - For each due row, compares the target plan's \`price_cents\` to the workspace's current plan price.
  - \`isUpgrade = target.price > current.price AND target.price > 0\`.
  - On upgrade → \`plan_status='none'\`, clears \`trial_ends_at\`, no trial granted. The existing dashboard layout redirect blocks the workspace via \`/dashboard/upgrade\`.
  - On downgrade/same-tier → existing behavior (trialing if eligible, else active).
  - Fires \`billing.plan_change_pending_payment\` (migration 083) on upgrade — copy is "add a payment method to activate" with the upgrade URL. Otherwise the existing \`billing.plan_changed\` template.
  - \`admin_audit\` metadata now carries \`is_upgrade\` + \`plan_status_after\` for the operator trail.

- **\`/dashboard/upgrade\` page**:
  - Detects the pending-operator-upgrade state via the most recently applied \`workspace_plan_changes\` row matching the workspace's current plan.
  - Renders an amber banner with the upgraded-to plan + effective date + "Add a payment method to start using the upgraded plan. Until then your workspace is paused on Free-plan limits."
  - Surfaces available credit (Stripe \`Customer.balance\`): "You have \$X.XX in credit — Stripe will apply it to your first invoice automatically." Display-only — Stripe handles the actual application on the first invoice.
  - When the new banner fires it suppresses the older \`fromSignup\` "Last step — add a payment method to activate your subscription." copy.

### Migrations

Both already applied to prod via MCP — no apply step needed at release. Migration files committed for the historical record.

### Smoke test path

1. Operator-schedule Bobby (Free) → Pro for a date a few days out. Confirm:
   - Customer email: "your plan will switch on [date]"
   - In-app banner on \`/dashboard/settings/billing\`: "Your plan switches to Pro on [date]"
2. Wait until the next \`:00\` UTC after the effective date, OR run the janitor manually via \`select public.call_saas_cron('/api/cron/apply-scheduled-plan-changes');\`. Confirm:
   - \`workspaces.plan='pro'\`, \`plan_status='none'\`, \`trial_ends_at=null\`
   - Customer email: "your plan changed — add a payment method to activate"
   - Bobby's dashboard redirects to \`/dashboard/upgrade?from=signup\`
   - Upgrade page shows the amber banner with the upgrade context
3. Click any plan → Stripe Checkout opens with card collection required → enter test card → on return, workspace flips to \`plan_status='active'\` via the existing webhook path.

If Bobby has a credit balance set (via \`POST /api/admin/workspaces/[id]/credit\`), the upgrade-page banner shows the dollar amount + Stripe auto-applies it to the first invoice.
