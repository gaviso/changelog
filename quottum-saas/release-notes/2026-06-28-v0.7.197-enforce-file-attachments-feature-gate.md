Hotfix surfacing from the v0.7.196 smoke: blocking \`file_attachments\` on a workspace via the operator console fired the customer email + popup but the upload UI stayed live and the upload endpoint accepted writes. Same broken pattern \`white_label\` had until v0.7.195 — feature in catalog, nobody consulting it.

### Fix

**Server** — \`/api/attachments/upload\`:
- \`featureGate('file_attachments', ...)\` called BEFORE the per-plan storage limit check. Returns standard 403 when blocked. This is the authority — even a curl bypassing the UI gets rejected.

**UI** — \`AttachmentsPanel\`:
- New \`featureEnabled\` prop. When false, the upload drop zone is replaced with "File attachments aren't available on this workspace. Upgrade your plan or reach out to support to enable uploads."
- Existing files still render with download + delete buttons — users can clean up storage after a block lands.

**Wiring** — quote/contract detail pages:
- \`hasFeature(wsId, 'file_attachments')\` runs in parallel with \`getB2Settings()\`, passes through.

### Cascade with v0.7.195/196

When the WelcomePopupPoller pushes a block-added popup to a user already viewing a quote page, the layout's \`router.refresh()\` now also re-renders the page with the gate in place — drop zone vanishes within ~30s of the operator clicking Block.

### Audit follow-up

Ran a parallel codebase audit during this release to find OTHER catalog features with the same unwired pattern. **Result: 26 catalog features have zero enforcement code anywhere** — including quote_builder, quote_pdf, contract_pdf, contract_lifecycle, public_quote_view, public_contract_view, webhooks, revenue_metrics, dashboard_analytics, and more. Operators can grant/block these via the editor, but the blocks have no effect. Findings will land as a follow-up batch.

No migration. Pure code.
