One-line cron fix. Unblocks the 5 pg_cron schedules (02:00, 02:30, 04:00, 08:00, 09:00 UTC) that had been failing since the v0.7.178 self-hosted-cron rollout with \`quottum_cron_secret not configured\`.

### Fixed

- **\`public.call_saas_cron()\`** reads the existing Vault secret \`cron_secret\` instead of \`quottum_cron_secret\`. Pure \`CREATE OR REPLACE\` (migration 078) — same signature, same SECURITY DEFINER, same schedule callers. Idempotent.

No code change, no API change. The cron endpoints (\`/api/cron/*\`) and their \`x-cron-secret\` validation are unchanged.

### Version-slot note

This took the v0.7.182 slot that the operator subscription-actions follow-ups (feature overrides, scheduled changes, welcome popup, \`AdminWorkspaceSubscription\` rebuild, \`billing.plan_changed\` / \`billing.scheduled_plan_change\` emails) were tentatively reserved for. Those become **v0.7.183+**.
