Completes the on-platform payment migration. Payment + payment method are now collected entirely on our own page via Stripe Elements — **hosted Stripe Checkout is retired**, and the inline flow is unconditional for **all** workspaces.

### Phase 3 — trial flows brought inline
- **Flow A** (add card during a DB-only trial) and **Flow B** (mid-trial plan switch) now open the inline `<PaymentElement>` step instead of redirecting, with carryover-aware copy (intent: new / add_card / switch).
- The `/api/billing/subscribe` route already handled these (a DB-only trial has no `stripe_subscription_id` so it skips the 409, and `resolveTrialDays` returns the carryover → `setup` mode / SetupIntent / no charge today). No functional backend change — documented only.

### Phase 4 — Checkout retired + flag removed (= default-on, end state)
- Deleted `/api/billing/checkout`, the Checkout success page, and `SuccessPollClient`.
- Removed the `inline_checkout` feature flag everywhere (catalog, the route's 404 gate, picker props). Inline no longer has an "off" path → it's the unconditional path for everyone (Phase 2's default-on, taken to its end state).
- Dropped the redirect branch from `UpgradeForm` + `AddPaymentMethodForTrial`.

### Kept
- The Stripe webhook (`checkout.session.completed` handler remains, simply won't fire), `resolveTrialDays`, `resolveOrCreateCustomer`.
- **Out of scope:** plan changes for an *already-active* Stripe subscription stay on the Stripe customer portal.

Built across 3 background agents (Phase 3 frontend + backend in parallel, Phase 4) with composed type-check + two zero-reference greps verifying the removal.
